# PDI v0.4.0

## Jarvis Tool Execution MVP

PDI v0.4.0 introduced the first real consumer execution boundary on top of
the v0.3 Read Pipeline.

Jarvis remains a separate package that depends on PDI's public Query Service.
PDI Core does not depend on Jarvis, and the existing Write Pipeline remains
unchanged.

## Added

- `ToolCall` as the immutable execution request.
- `ToolResult` and `ToolError` as stable execution outcomes.
- `Tool` Protocol for the minimal Tool contract.
- `ToolRegistry` with explicit duplicate and unknown Tool handling.
- `JarvisApplication` for Tool lookup, execution, logging, and error
  containment.
- `create_jarvis_application(settings)` as the explicit Composition Root.
- `ListAssetsTool` backed by `QueryService.list_assets()`.
- `GetAssetTool` backed by `QueryService.get_asset(asset_id)`.

## Execution Path

```text
ToolCall
  ↓
JarvisApplication
  ↓
ToolRegistry
  ↓
Tool
  ↓
QueryService
  ↓
QueryRepository
  ↓
PostgreSQLRepository
  ↓
PostgreSQL
```

## Validation

- Full test suite: `81 passed`.
- Real PostgreSQL integration path validated.
- Asset summaries remain deterministically ordered.
- Asset, Blob, Source, and Source-to-Blob relationships remain available
  through immutable Read Models.
- ORM objects, Sessions, Engines, and Repository instances do not cross the
  Tool result boundary.
- Unknown Tools, invalid arguments, missing Assets, and unexpected failures
  return stable error codes.

No database schema or migration change was required for this release.
